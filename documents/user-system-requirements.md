## 一、背景与目标

当前博客已接入评论系统（Giscus），但评论身份完全依赖 GitHub 账号。为了支持更完整的互动能力（站内用户、权限控制、后续点赞/阅读量/通知等），需要建设统一用户系统。

本阶段先完成需求定义，目标是为后续实现提供清晰边界与可执行规格。

### 1.1 建设目标
- 提供站内统一身份（登录态、用户资料、权限）。
- 与评论能力衔接，支持“可追溯的站内用户”。
- 为后续互动模块（点赞、收藏、通知、消息）预留扩展位。
- 不破坏现有内容系统（MDX 文件发布）与 SEO 性能基线。

### 1.2 设计原则
- 先做最小闭环（MVP），再扩展社交能力。
- 认证方案优先选择成熟托管/标准库，避免自造轮子。
- 安全优先于功能数量（会话安全、权限隔离、审计可追踪）。

## 二、范围定义（Scope）

### 2.1 V1 必做范围（MVP）
- 用户注册/登录/退出。
- 第三方登录（优先 GitHub，兼容 Google 扩展）。
- 基础用户资料（昵称、头像、简介）。
- 角色权限（`user` / `admin`）。
- 会话管理（Cookie Session，过期与续期策略）。
- 路由级鉴权（后台发布入口、管理接口保护）。
- 评论身份映射（评论时可识别站内用户身份）。

### 2.2 V1 不做范围（Out of Scope）
- 复杂社交关系（关注/粉丝）。
- 站内私信。
- 多租户组织体系。
- 风控高级策略（设备指纹、行为风控引擎）。

### 2.3 V1.5+ 预留范围
- 用户通知中心（评论回复提醒）。
- 点赞/收藏与用户行为历史。
- 邮件验证码登录（Passwordless）。
- 风险登录告警与二次验证（2FA）。

## 三、用户系统需求分解（功能拆解）

## 模块 A：认证（Authentication）

### A1 登录方式
- GitHub OAuth 登录（首发必做）。
- 邮箱魔法链接（可选，作为后续扩展位）。

### A2 登录态管理
- 服务端会话校验（App Router 下的服务端鉴权）。
- 未登录用户访问受限路由时，重定向到登录页。

### A3 退出登录
- 前端显式退出入口。
- 退出后会话立即失效，敏感页面不可回退访问。

## 模块 B：账户与资料（User Profile）

### B1 基础资料
- 字段：`name`、`avatar`、`bio`、`website`。
- 昵称限制：2~30 字符，去除危险脚本内容。

### B2 资料编辑
- 仅用户本人可编辑自己的资料。
- 管理员可在后台查看/编辑/禁用用户（V1 后段或 V1.5）。

## 模块 C：权限与后台保护（Authorization）

### C1 角色模型
- `user`：普通用户，仅能管理自己的资料与互动数据。
- `admin`：可访问发布后台、用户管理、评论管理。

### C2 权限策略
- 采用 RBAC（基于角色的访问控制）。
- 后端 API 必须二次鉴权，前端仅作为显示层限制。

## 模块 D：评论系统衔接（Comment Integration）

### D1 兼容策略
- 短期：保留 Giscus，站内仅展示“当前登录用户信息”与后续扩展能力。
- 中期：可切换为站内评论（数据库存储），复用本用户系统。

### D2 数据一致性
- 每条评论必须可关联到 `userId`（若未来切换站内评论）。
- 评论删除与用户禁用策略需有审计记录。

## 模块 E：安全与审计（Security & Audit）

### E1 安全基线
- OAuth 回调校验、CSRF 防护、敏感操作防重放。
- 会话 Cookie 使用 `HttpOnly`、`Secure`、`SameSite=Lax/Strict`。

### E2 审计事件
- 登录成功/失败、角色变更、账户禁用、后台关键操作记录。

## 四、角色与使用场景

| 角色 | 核心诉求 | 关键动作 |
|---|---|---|
| 访客（Guest） | 阅读文章、浏览站点 | 浏览、跳转、分享 |
| 注册用户（User） | 登录身份、参与互动 | 登录、编辑资料、发表评论（未来站内） |
| 管理员（Admin） | 运营与管理 | 发布文章、管理用户、审核评论（未来） |

## 五、业务流程（核心）

### 5.1 首次登录流程（OAuth）
1. 用户点击“GitHub 登录”。
2. 跳转 OAuth 授权页并确认权限。
3. 回调后创建/更新用户记录。
4. 建立会话，回到来源页面。

### 5.2 鉴权访问流程
1. 用户访问受限路由（如 `/admin/publish`）。
2. 中间件/服务端检查会话与角色。
3. 未登录或无权限则重定向到登录/无权限页。

### 5.3 资料更新流程
1. 用户进入个人设置页。
2. 修改资料并提交。
3. 后端校验字段合法性后写入数据库。
4. 返回成功状态并刷新展示信息。

## 七、数据模型（建议）

> 推荐采用 `Auth.js + Prisma + PostgreSQL`。  
> 可部署于 Vercel + Supabase / Neon / Vercel Postgres。

### 7.1 核心表

#### `User`
| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | `string` | PK | 用户唯一标识 |
| `email` | `string` | Unique, Nullable | 邮箱（OAuth 可为空后补） |
| `name` | `string` | Nullable | 昵称 |
| `avatar` | `string` | Nullable | 头像 URL |
| `bio` | `string` | Nullable | 简介 |
| `role` | `enum` | Default `user` | `user/admin` |
| `status` | `enum` | Default `active` | `active/disabled` |
| `createdAt` | `datetime` |  | 创建时间 |
| `updatedAt` | `datetime` |  | 更新时间 |

#### `Account`（OAuth 账号映射）
| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | `string` | PK | 记录 ID |
| `userId` | `string` | FK -> User | 归属用户 |
| `provider` | `string` |  | 如 `github` |
| `providerAccountId` | `string` | Unique(provider+id) | 第三方账号 ID |
| `access_token` | `string` | Nullable | 第三方访问令牌 |

#### `AuditLog`（建议新增）
| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | `string` | PK | 日志 ID |
| `actorUserId` | `string` | Nullable | 操作人 |

| `targetType` | `string` |  | 作用对象类型 |
| `targetId` | `string` |  | 作用对象 ID |
| `meta` | `json` | Nullable | 扩展信息 |
| `createdAt` | `datetime` |  | 记录时间 |

## 八、接口需求（V1）

| 接口 | 方法 | 描述 | 权限 |
|---|---|---|---|
| `/api/auth/signin` | `POST` | 发起登录 | 公开 |
| `/api/auth/signout` | `POST` | 退出登录 | 登录用户 |
| `/api/user/profile` | `GET` | 获取当前用户资料 | 登录用户 |
| `/api/user/profile` | `PATCH` | 更新当前用户资料 | 登录用户 |
| `/api/admin/users` | `GET` | 用户列表查询 | `admin` |
| `/api/admin/users/:id/role` | `PATCH` | 调整用户角色 | `admin` |
| `/api/admin/users/:id/status` | `PATCH` | 启用/禁用用户 | `admin` |

接口约束：
- 所有写操作返回统一错误结构：`{ code, message, requestId }`。
- 鉴权失败返回 `401`，权限不足返回 `403`。
- 管理接口必须记录审计日志。

## 九、非功能需求

### 9.1 安全
- 不存储明文密码（若未来支持密码登录，必须强哈希如 Argon2/Bcrypt）。
- 关键接口限流（登录、资料更新、后台操作）。
- 统一输入校验（Zod 或同类方案），防止注入与 XSS。

### 9.2 性能
- 鉴权中间件开销可控，不显著影响页面 TTFB。
- 用户资料读取 P95 < 300ms（同区域部署）。

### 9.3 可用性
- OAuth 回调异常可回退并提示可理解错误信息。
- 会话过期后有友好重登引导，不出现空白页。

### 9.4 可维护性
- 认证、用户资料、权限逻辑分层组织。
- 审计日志结构稳定，方便后续接入监控系统。

## 十、验收标准（DoD）

- 可通过 GitHub 登录并创建站内用户记录。
- 登录后可访问个人设置页并更新资料。
- 未登录访问受限页面会被拦截并跳转。
- `user` 无法访问管理员页面，`admin` 可访问。
- 管理员变更角色/状态后，权限即时生效。
- 核心鉴权接口具备基础安全头与错误处理。
- 关键管理操作可在审计日志中追踪。

## 十一、实施里程碑（建议）

### M1：认证骨架（2~3 天）
- 引入 Auth.js，完成 GitHub 登录闭环。
- 建立 User/Account/Session 表与迁移脚本。

### M2：资料与权限（2 天）
- 实现个人资料页与资料接口。
- 加入 RBAC 与路由保护。

### M3：后台与审计（2 天）
- 管理员用户管理页（最小可用）。
- 审计日志写入与查询。

### M4：评论联动（1~2 天）
- 统一评论区身份显示逻辑。
- 预留站内评论切换接口。

## 十二、风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| OAuth 配置错误（回调域名） | 无法登录 | 增加配置检查清单与预发验证 |
| 角色越权 | 安全问题 | 后端强制鉴权 + 审计 + 测试覆盖 |
| 数据库引入后复杂度提升 | 运维成本增加 | 先选托管 Postgres，保持 schema 简洁 |
| 与现有 Giscus 体验割裂 | 用户感知不一致 | 明确短期兼容策略，逐步迁移 |

## 十三、待确认事项（必须决策）

1. 评论系统是否继续长期使用 Giscus，还是切换站内评论？
2. 登录方式首发是否仅 GitHub，是否同步接入 Google？
3. 是否在 V1 即开放“用户注册入口”，还是仅 OAuth 登录即注册？

5. 是否需要对接邮件服务（登录提醒、通知）？

## 十四、结论

用户系统建议采用“**OAuth 登录 + 站内用户模型 + RBAC + 审计**”的最小闭环方案，先支撑评论身份与后台权限，再扩展互动能力。  
该文档可作为阶段八后续开发的直接执行规范。
