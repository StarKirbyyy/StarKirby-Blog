## 一、背景与目标

当前站点功能已较完整（发布、编辑、评论、用户、云端存储），但前台视觉风格偏“通用技术博客”，个性辨识度不够。

本次目标是对 Web UI 做一次全量重构，参考 `blog.azifan.club` 的视觉语言，形成“有氛围、强个性、可长期维护”的博客外观，同时保持已有业务能力不回退。

### 1.1 核心目标
- 在不破坏现有功能链路的前提下，重构所有前台核心页面视觉。
- 建立可复用设计 Token（色彩、圆角、阴影、层级、动效）与组件规范。
- 强化首页与文章页的“沉浸感、叙事感、可读性”。
- 保持移动端可用与性能达标，不做“只好看但不可用”的方案。

### 1.2 非目标（本轮不做）
- 不复制目标站点源码、图片、文案、品牌元素。
- 不更换内容模型（Post / Comment / User）与核心业务 API。
- 不引入多人协同编辑等新业务功能。

## 二、参考风格分析（需求输入）

以下为基于可抓取页面与其主题文档提炼出的风格特征，用于“风格映射”，不是 1:1 克隆：

- 封面主导：首页有明显视觉封面与信息栏组合。
- 信息栏人格化：头像、签名、社交入口聚合成高识别模块。
- 半透明层与层次感：导航/卡片具备玻璃感与浮层关系。
- 卡片化文章区：图文卡片、圆角、阴影、轻动效。
- 氛围型背景：可切换背景（含随机图能力）增强页面生命力。
- 细节交互密度高：悬停、滚动、切换反馈丰富，但不影响阅读主流程。

### 2.1 风格迁移原则
- 保留“氛围 + 人格化 + 卡片化”三大特征。
- 结合 `starkirby` 品牌语义重做配色与字体，不直接照搬色值。
- 优先保证“文章可读性”和“信息层级”高于装饰效果。

## 三、范围定义（Scope）

### 3.1 V1 必做范围（全量前台）
- 全局：`Header / Footer / 全局背景 / 主题切换 / 页面容器`。
- 首页：封面区、信息栏、社交区、最新文章区、分页/跳转入口。
- 列表页：`/posts`、`/tags`、`/tags/[tag]`。
- 详情页：`/posts/[slug]`（含封面、目录、上下篇、评论区）。
- 其他页：`/about`、`/projects`、`/404`、`/login`。
- 保持后台页面功能可用，视觉做“轻量统一”（不过度装饰）。

### 3.2 V1 不做范围
- 后台管理台完整视觉重设计系统（仅基础统一）。
- 自定义主题编辑器（用户可视化配色器）。
- 动态背景视频等重资源特效。

## 四、信息架构与页面重构要求

## 模块 A：全局框架

### A1 顶部导航（Header）
- 采用半透明悬浮导航（支持 `backdrop-blur`）。
- 桌面端展示主导航 + 社交快捷入口 + 主题切换。
- 移动端折叠菜单与桌面信息一致，不出现功能缺失。

### A2 全局背景层
- 提供“静态背景 + 随机背景”双模式。
- 背景仅作为氛围层，不得降低正文可读性（正文区需独立对比层）。
- 支持开关，弱网场景自动降级为轻量背景。

### A3 页脚（Footer）
- 提供备案、版权、社交、可选一言区域。
- 排版支持“居中 / 双栏”两种可配置形态（默认居中）。

## 模块 B：首页（重点）

### B1 Hero 封面区
- 首屏包含：背景图、站点名、标语、向下引导。
- 提供“随机换背景”触发按钮（可选文案）。

### B2 信息栏（Profile Panel）
- 头像、签名、简述、社交图标统一在一个信息卡片内。
- 样式支持“分离式 / 合并式”两种布局（默认分离式）。

### B3 文章区域
- 使用图文卡片样式，支持左右交替布局（桌面）与单列布局（移动）。
- 卡片展示：标题、日期、阅读时长、摘要、标签、封面。
- 首屏加载优先保证文字可见，图片渐进加载。

## 模块 C：文章列表与详情

### C1 文章列表页 `/posts`
- 与首页文章卡片保持统一视觉语言。
- 分页样式重构为胶囊化按钮，明确当前页状态。

### C2 文章详情页 `/posts/[slug]`
- 头部区强化：标题、描述、meta 信息层级更清晰。
- 正文容器可读性优先：控制行宽、段间距、代码块对比度。
- 目录（TOC）采用轻浮层/吸附布局，滚动时具备状态反馈。

### C3 详情辅助区
- 上下篇导航改为视觉卡片，增强可点击性。
- 评论区与正文视觉解耦，避免“全页同质化”。

## 模块 D：其他页面一致性

### D1 标签/关于/项目/登录/404
- 统一为同一视觉框架（背景层、卡片、字体层级、按钮体系）。
- 各页面保持个性区块，但组件风格必须统一。

### D2 后台页面（轻量）
- 保持表单可用性与信息密度，不引入过多装饰。
- 统一基础按钮、输入框、提示样式。

## 五、视觉系统规范（Design Tokens）

### 5.1 色彩策略
- 建议采用“三层色彩”：
  - 基础层：背景/文字/边框（保证对比度）。
  - 品牌层：主强调色（按钮、链接、焦点）。
  - 氛围层：封面渐变、卡片高光、阴影色。
- 明暗主题均需完整 token，不允许“只做亮色”。

### 5.2 字体策略
- 中文正文字体：可读优先（现有 MiSans 可保留）。
- 标题字体：允许更有个性的展示字体，但必须可降级。
- 代码字体继续独立 monospace。

### 5.3 形状与层级
- 圆角分级：`sm/md/lg/xl/full` 统一管理。
- 阴影分级：卡片、悬浮、弹层分离。
- 边框透明度与毛玻璃强度纳入 token 管理。

### 5.4 动效规范
- 页面切换、卡片 hover、按钮反馈统一时长和缓动。
- 禁止高频炫技动效影响阅读（动画默认 150~300ms）。
- 尊重 `prefers-reduced-motion`。

## 六、技术实现约束

### 6.1 技术栈约束
- 继续使用 `Next.js App Router + Tailwind v4`。
- 优先通过 `globals.css` + 语义 class 重构，不新增重型 UI 框架。
- 可引入轻量动画库，但需通过性能评估后再接入。

### 6.2 代码组织约束
- 组件按职责拆分：`layout / pages / cards / decorations / shared`。
- 避免在页面文件堆积大段样式字符串。
- 新样式变量优先放入全局 token，避免组件内硬编码色值。

### 6.3 兼容与降级
- 桌面（>=1024）、平板（768~1023）、移动（<768）必须分别验收。
- 背景图失败时有纯色降级，不影响内容可读。
- 浏览器最低要求：Chrome/Edge/Safari 最新两个主版本。

---

## 七、非功能指标（必须达标）

- 性能：
  - 首页 LCP（4G 模拟）目标 <= 2.8s。
  - CLS <= 0.1。
  - 首屏关键区域无布局抖动。
- 可访问性：
  - 文本与背景对比度满足 WCAG AA。
  - 所有可交互元素有可见焦点。
- SEO：
  - 不破坏现有 Metadata、OG、RSS、Sitemap。
  - 不因重构导致主要页面不可索引。

---

## 八、验收标准（DoD）

- 视觉验收：
  - 首页、列表、详情与目标风格在“气质层面”一致（封面感、卡片感、玻璃层感）。
  - 整站视觉语言统一，无明显“旧样式残留”。
- 交互验收：
  - 背景切换、导航、分页、目录、评论区域行为稳定。
- 业务验收：
  - 发布、编辑、评论、用户登录流程无功能回归。
- 质量验收：
  - `pnpm lint`、`pnpm build` 全通过。
  - 关键页面移动端可正常阅读与操作。

## 九、里程碑拆解（建议）

### M1：视觉 Token 与全局框架（1~2 天）
- 重构 `globals.css` token 体系。
- 重做 Header/Footer 与全局容器层级。

验收：
- 全站基础样式统一，旧 token 不再新增依赖。

### M2：首页重构（1~2 天）
- 完成封面区、信息栏、随机背景、文章区卡片。

验收：
- 首页达到新视觉基调，移动端无排版断裂。

### M3：列表页与详情页（1~2 天）
- `/posts`、`/posts/[slug]`、TOC、上下篇、评论区重构。

验收：
- 文章阅读链路完整，视觉与首页一致。

### M4：其余页面与后台轻量统一（1 天）
- `/tags`、`/about`、`/projects`、`/login`、`/404` 统一风格。
- 后台基础控件样式对齐。

验收：
- 路由间切换无明显风格割裂。

### M5：动效与性能优化（0.5~1 天）
- 动效节奏统一、图片策略优化、降级策略完善。

验收：
- Lighthouse 指标满足目标线。

### M6：回归测试与上线（0.5~1 天）
- 回归核心业务、修复尾部问题、上线发布。

验收：
- 功能无回归，可稳定部署。

## 十、风险与待确认项

### 10.1 主要风险
- 视觉装饰过重导致可读性下降。
- 背景图和特效影响首屏性能。
- 只重构首页导致内页风格不统一。

### 10.2 必须确认（开始开发前）
- 是否采用“二次元/ACG 氛围”作为明确品牌方向。
- 背景图来源策略：本地静态图 / OSS 图库 / 外部 API。
- 默认主题：亮色或暗色优先。
- 首页信息栏文案（签名/标语）是否支持后台配置。
